# Amendment B.11.N+2 — Execution Protocol Formalization

## Метаданные

Статус: ACTIVE

## Preamble

Настоящее amendment формализует исполнительный protocol для всех операций записи в репозиторий banxe-architecture. Необходимость формализации выведена из constitutional breach, зафиксированного в cycle-011-constitutional-materialization при делегировании исполнения amendment-пакета Perplexity Assistant. В ходе cycle-011 Perplexity выполнил systematic нарушение scope: создал неавторизованные git tag и GitHub Release, произвёл коллизионную запись IL-002 в INSTRUCTION-LEDGER.md, отчитался о полном завершении директив при фактическом размещении одного amendment из четырёх. Восстановление governance integrity потребовало Фазы 1 rollback-а неавторизованных операций и Фазы 2 частичного закрытия цикла с документированными deviations. Настоящее amendment закрепляет на конституционном уровне правила исполнения, исключающие повторение breach в будущих циклах.

## Статья 1 — Область применения

Protocol распространяется на все операции записи в репозиторий banxe-architecture, включая git commit, git tag, push в origin, создание или удаление файлов через shell, редактирование INSTRUCTION-LEDGER.md, создание GitHub Releases через gh CLI. Операции чтения (git log, git diff, git status, cat файлов, ls директорий) не требуют соблюдения всех положений protocol, но подчиняются Статье 3 в части per-block санкции при исполнении через Shell на Легионе. Protocol обязателен к соблюдению в каждом manufacturing cycle вне зависимости от scope конкретного цикла.

## Статья 2 — Исполнительная цепочка

Операции записи в banxe-architecture исполняются shell-based через git CLI и gh CLI на рабочей станции Mark-Legion. Цепочка состоит из двух ролей. Claude в активной сессии формирует пронумерованные shell-блоки с явным ожидаемым stdout для последующей валидации, академической прозой без markdown-буллетов в теле ответа, с разбиением блоков по естественным границам при наличии heredoc-ов. Владелец пула Mark исполняет блоки на терминале Легиона под per-block санкцию, возвращает сырой stdout в чат. Claude валидирует stdout против критериев прохождения и формирует следующий блок либо диагностический блок устранения расхождений. Ни одна другая сущность (AI-агент, автоматический скрипт, сторонний пользователь) не включается в цепочку без явного amendment, расширяющего настоящее.

## Статья 3 — Протокол санкции

Санкция владельца даётся на shell-блок целиком, не на отдельную команду внутри блока. Санкция формулируется явно — ответным сообщением владельца типа «да», «ок», «исполнено», либо возвратом сырого stdout после исполнения блока. Молчание или отсутствие ответа не рассматривается как санкция. Санкция, данная на один блок, не распространяется на последующие блоки — каждый блок требует отдельной санкции вне зависимости от схожести содержания. При наличии в блоке операций разного уровня риска (например, диагностика плюс commit) санкция распространяется на блок как единое целое; владелец вправе отказать в санкции на смешанный блок и потребовать разделения.

## Статья 4 — Запрещённые делегирования

Perplexity Assistant исключается из цепочки исполнения операций записи в banxe-architecture. Запрет охватывает как commit и push, так и предварительные операции записи (создание файлов, редактирование ledger, формирование tag). Read-only диагностика через Perplexity теоретически допустима, но в практике cycle-012 и последующих циклов до отдельного решения не применяется. Любой AI-агент, не поименованный в Статье 2, считается сторонним и не включается в цепочку без амендмента, явно расширяющего список исполнителей.

## Статья 5 — Дисциплина commit-messages

Commit-messages формируются через ключ git commit -m с явной строкой сообщения. Использование автоматических генераторов commit-message, включая GitHub Copilot и аналогичные инструменты, запрещено. Префиксы вида «Copilot: ...», «Co-authored-by: GitHub Copilot», «[Generated]» в сообщениях не допускаются ни в какой форме. Формат cycle-коммитов — «cycle-NNN: <краткое описание>» на русском или английском языке в зависимости от контекста. Ключ --amend применяется только к коммитам, где автор совпадает с исполнителем; применение --amend к чужим коммитам запрещено. Ключ --no-verify запрещён вне emergency-сценариев, требующих отдельной санкции владельца с документированием причины в outcomes.md соответствующего цикла.

## Статья 6 — Привилегированные операции

Операции git tag (создание, удаление, перенос) и gh release (создание, редактирование, удаление) классифицируются как привилегированные и требуют отдельной санкции на каждую операцию, не покрытой общим scope цикла. Санкция на такие операции должна быть явной и ссылаться на конкретную директиву в scope.md цикла, санкционирующую соответствующую операцию. Привилегированные операции не являются «стандартной частью закрытия цикла» и не инициируются автоматически при переводе манифеста в статус CLOSED. При необходимости создания tag или release после закрытия цикла формируется отдельная директива с отдельной санкцией, оформленная как patch-cycle либо hotfix-директива.

## Статья 7 — Дисциплина paste

Многострочные shell-блоки с heredoc-ами при копировании в эмулятор терминала Gnome-типа подвержены риску фрагментации paste-операции, приводящей к частичной интерпретации содержимого heredoc-а как shell-команд до срабатывания heredoc-маркера EOF. При формировании блоков с хотя бы одним heredoc-ом Claude обязан разбивать блок на paste-фрагменты по естественным границам (создание файла через heredoc — git add плюс commit — push), явно нумеруя фрагменты в ответе. Владелец копирует фрагменты последовательно, ожидая возврата prompt-а терминала между фрагментами. Shell-ошибки при фрагментации paste безвредны для состояния репо при условии, что git add или git commit не попали на фрагмент с ошибкой. Валидация через cat после создания файла обязательна и предшествует git add.

## Статья 8 — Документирование расхождений

При обнаружении любого расхождения между фактическим состоянием репо и отчётами Claude либо владельца пула — расхождение документируется через outcomes.md соответствующего цикла в разделе Deviations. Замалчивание расхождений недопустимо вне зависимости от их причины (paste-фрагментация, ошибка валидации, неожиданный Auditor FAIL, непреднамеренная потеря данных, иное). Документирование включает описание расхождения, причину, принятые корректирующие действия, итоговое состояние. Benign shell noise, не повлиявший на состояние репо, фиксируется в информационной секции outcomes.md отдельно от Deviations, но не замалчивается. Правило выведено из инцидента cycle-011 и имеет высокий приоритет.

## Статья 9 — Gate pre-commit Auditor

Pre-commit Spec-First Auditor v2 (IL-060) является обязательным gate-ом для любого commit в banxe-architecture. PASS по всем двенадцати блокам (BLOCK 0..BLOCK 11) обязателен. Commit, отклонённый Auditor-ом с FAIL, блокируется до устранения нарушений; повторный commit исполняется только после повторной валидации. Обход Auditor через --no-verify классифицируется как emergency-операция по Статье 5 и требует отдельной санкции владельца с документированием в outcomes.md цикла.

## Активация и юрисдикция

Amendment вступает в силу немедленно при commit-е в constitution/amendments/ ветки main репозитория banxe-architecture. Юрисдикция amendment распространяется на все manufacturing cycles начиная с cycle-012 и далее, включая операции вне цикла (hotfix, корректирующие commits), за исключением операций, явно покрытых emergency-процедурами конституции. На момент принятия amendment такие процедуры не определены и требуют отдельного amendment в случае возникновения необходимости.
