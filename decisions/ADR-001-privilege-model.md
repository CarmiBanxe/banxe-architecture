# ADR-001: Модель привилегий — разработчик vs оператор-дублёр

**Статус:** ACCEPTED  
**Дата:** 2026-04-05  
**Автор:** Moriel Carmi (CEO)

---

## Контекст

Система Banxe AI Bank включает:
- AI агентов (OpenClaw, compliance stack) — настраиваются разработчиком
- Операторов-людей (MLRO, Compliance Officer) — работают с кейсами в реальном времени

Возник вопрос: что именно может оператор изменять через свой терминал (Telegram-бот, Marble UI)?

## Проблема

Если граница не зафиксирована явно, существует риск:
1. Оператор случайно меняет поведение агента через операторский интерфейс
2. Разработчик не понимает, что оператор может влиять на систему
3. FCA audit trail не разделяет "решение агента" и "решение оператора"

## Решение

Жёсткое разделение на два непересекающихся домена:

**DEVELOPER** — настраивает систему (SOUL.md, thresholds, training pipeline)  
**OPERATOR** — работает внутри системы (HITL decisions, case management)

Оператор не может менять правила — только применять их к конкретным кейсам.

## Последствия

- Telegram-бот и Marble UI — read/decide only для оператора
- Все изменения конфигурации требуют developer-side deploy
- FCA audit trail: `decision_source` = "agent" | "operator" | "developer"
- Удалена неопределённость: оператор знает точно, что он может
