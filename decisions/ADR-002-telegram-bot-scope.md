# ADR-002: Telegram-бот — область применения

**Статус:** ACCEPTED  
**Дата:** 2026-04-05  
**Автор:** Moriel Carmi (CEO)

---

## Контекст

Telegram-бот `@mycarmi_moa_bot` уже развёрнут и используется.  
Поступил вопрос: должен ли он служить банковским приложением для клиентов (платежи, KYC, баланс)?

## Два варианта использования

| Вариант | Описание | Статус |
|---------|----------|--------|
| **Терминал оператора** | Алерты MLRO, HITL-решения, кейсы Marble | CURRENT — Active |
| **Клиентское приложение** | Платежи, баланс, KYC onboarding для клиентов банка | DEFERRED |

## Решение

Текущий Telegram-бот = **терминал оператора-дублёра**. Только это.

Telegram как банковское приложение для клиентов — **отдельная инсталляция**, отдельный OpenClaw instance, отдельный проект. Разработка откладывается до:
1. Завершения Phase 3 KYC (PassportEye + DeepFace)
2. Получения необходимых FCA permissions для клиентских операций
3. Дизайна UX клиентского flow

## Причины разделения

- Разные security модели: оператор (trusted internal) vs клиент (untrusted external)
- Разные compliance требования: HITL vs full KYC/AML
- Смешивание → путаница в audit trail и роутинге агентов
- Риск: клиент получит доступ к internal compliance данным

## Последствия

- Текущий бот: scope locked — только оператор-дублёр
- Клиентский бот: ADR обязателен перед началом разработки
- `DEFERRED-PROJECTS.md` зафиксирует статус
